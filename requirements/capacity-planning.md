แน่นอนครับ นี่คือคำถามที่สำคัญที่สุดในการทำ Capacity Planning

ในฐานะ Master Architect ผมได้วิเคราะห์เอกสาร `requirements/business.md` ที่คุณให้มา เพื่อประเมินค่า RPS (Requests Per Second) ที่ระบบของเรา **ต้อง** รองรับให้ได้ในช่วงเวลาที่พีคที่สุดครับ

### บทสรุปการประเมิน

จากการวิเคราะห์ Flow การทำงานที่ระบุในเอกสาร โดยเฉพาะส่วนของ Waiting Room ผมประเมินว่า RPS ที่ **Booking Service (`POST /api/bookings`)** ของเราต้องรับมือให้ได้จะอยู่ที่ประมาณ **2,000 - 5,000 RPS** ในช่วงเวลาที่พีคที่สุด (Peak Burst) ครับ

---

### หลักการและข้อสมมติฐานในการประเมิน

การคำนวณนี้อิงตามข้อมูลและข้อสมมติฐานที่สมเหตุสมผลที่สุดจากเอกสาร `requirements/business.md`:

1.  **จุดที่ต้องโฟกัส (The Hotspot):** เราจะคำนวณ RPS ที่เข้ามายัง `Booking Service` เท่านั้น เพราะนี่คือบริการที่ต้องรับภาระงานหนักที่สุด ไม่ใช่จำนวนคนทั้งหมด 1 ล้านคนที่รออยู่ข้างนอก
2.  **กุญแจสำคัญคือ Waiting Room:** เอกสารระบุว่า *"Waiting Room ทยอยปล่อยคนเข้าเว็บทีละ 10,000 คน"* นี่คือตัวเลขที่สำคัญที่สุดในการคำนวณของเรา
3.  **พฤติกรรมผู้ใช้ (User Behavior):** หลังจากผู้ใช้ 10,000 คนถูกปล่อยเข้ามาแล้ว พวกเขาจะไม่กดจองพร้อมกันในมิลลิวินาทีเดียวกัน จะมีช่วงเวลาสั้นๆ ที่ใช้ในการมองหาที่นั่งและตัดสินใจกดจอง เราจะเรียกช่วงเวลานี้ว่า **"หน้าต่างเวลาที่ผู้ใช้ลงมือ" (Action Window)**

---

### การคำนวณ RPS ในช่วงเวลาที่พีคที่สุด (Peak Burst RPS)

เราจะคำนวณโดยใช้สูตร: `RPS = (จำนวนผู้ใช้ที่ถูกปล่อยเข้ามา) / (Action Window)`

#### สถานการณ์สมจริง (Realistic Case): **Action Window = 5 วินาที**
*   **สมมติฐาน:** หลังจากหน้าเว็บโหลดเสร็จ ผู้ใช้ส่วนใหญ่จะใช้เวลาคิดและคลิกจองครั้งแรกภายใน 5 วินาทีแรก
*   **การคำนวณ:** `10,000 users / 5 seconds` = **2,000 RPS**

#### สถานการณ์เลวร้ายที่สุด (Worst-Case Spike): **Action Window = 2 วินาที**
*   **สมมติฐาน:** ผู้ใช้เตรียมพร้อมมาอย่างดีและคลิกจองอย่างรวดเร็วที่สุด ทำให้ request ทั้งหมดถูกส่งเข้ามาในช่วงเวลาแค่ 2 วินาที
*   **การคำนวณ:** `10,000 users / 2 seconds` = **5,000 RPS**

### RPS เฉลี่ยต่อเนื่อง (Sustained RPS)

หลังจาก Burst แรกผ่านไป ผู้ใช้ประมาณ 9,500 คน (สมมติว่ามีที่นั่ง 500 ที่) จะได้รับ `409 Conflict` และจะพยายามกดจองซ้ำๆ การกระทำนี้จะสร้างภาระงานต่อเนื่องให้กับระบบ ซึ่งจะมีค่า RPS ที่สูงใกล้เคียงกับช่วง Peak Burst ต่อไปอีกระยะหนึ่งจนกว่าที่นั่งจะหมด

---

### คำแนะนำของ Architect

**ระบบของเราต้องถูกออกแบบมาให้ "รอด" จาก Peak Burst ให้ได้โดยไม่ล่ม** และสามารถทำงานต่อเนื่องภายใต้ Sustained RPS ได้

ดังนั้น เป้าหมายในการทำ Performance Test ของเราควรตั้งไว้ที่ **การรองรับให้ได้ 5,000 RPS อย่างมีเสถียรภาพ** ซึ่งเป็นการเตรียมพร้อมสำหรับสถานการณ์ที่เลวร้ายที่สุดตามข้อมูลที่เรามี

*   **เป้าหมาย 10k RPS ที่คุณตั้งไว้ก่อนหน้านี้ถือว่ายอดเยี่ยมมาก** และเป็นเป้าหมายที่ดีหากต้องการสร้างระบบที่มี Headroom เหลือเฟือ
*   **แต่เป้าหมายขั้นต่ำที่ "ต้องทำให้ได้"** ตาม requirement ที่เรามี คือ **5,000 RPS** ครับ

หากการทดสอบ 2,000 VUs ของเราผ่านไปได้ด้วยดี เราควรจะปรับสคริปต์ k6 ไปใช้ `constant-arrival-rate` executor เพื่อทดสอบที่เป้าหมาย 5,000 RPS ต่อไปครับ

---

ยอดเยี่ยมมากครับที่คุณผ่านการทดสอบ 2,000 VUs มาได้ นั่นแสดงว่าสถาปัตยกรรมของเราแข็งแกร่งและพร้อมสำหรับความท้าทายขั้นต่อไป

ในฐานะ Master Architect ผมได้คำนวณจำนวน VUs โดยประมาณที่คุณต้องใช้เพื่อสร้างภาระงาน (RPS) ตามเป้าหมายที่คุณต้องการแล้วครับ โดยอิงจากข้อมูลการทดสอบที่ผ่านมา

### หลักการคำนวณ (The "Why")

การคำนวณนี้อยู่บนพื้นฐานของ **กฎของลิตเติ้ล (Little's Law)** ซึ่งเป็นหลักการพื้นฐานใน Performance Engineering:

`จำนวนผู้ใช้พร้อมกัน (VUs) = อัตราการเข้า (RPS) × เวลาที่ใช้ในแต่ละรอบ (Iteration Duration)`

จากผลการทดสอบที่ผ่านมาของเรา:
*   **Average Iteration Duration:** อยู่ที่ประมาณ **2.0 วินาที**
    *   ค่านี้มาจาก `iteration_duration.............: avg=2.01s` ในผลลัพธ์ k6
    *   ซึ่งสมเหตุสมผลมาก เพราะในสคริปต์ `load-test.js` ของเรา มี `sleep(randomIntBetween(1, 3))` ซึ่งมีค่าเฉลี่ยคือ 2 วินาที บวกกับเวลาตอบสนองของ API ที่เร็วมาก (หลักมิลลิวินาที)

ดังนั้น เราสามารถคำนวณ VUs ที่ต้องการได้จาก:
`VUs = Target RPS × 2.0`

**ตัวอย่าง:**
*   สำหรับ 2,500 RPS: `2500 × 2.0 = 5000 VUs`
*   สำหรับ 10,000 RPS: `10000 × 2.0 = 20000 VUs`

### **คำเตือนที่สำคัญมากจาก Architect**

อย่างที่เราคุยกันไปครั้งที่แล้ว **คุณไม่สามารถรัน VUs จำนวนมากขนาดนี้บน PC เครื่องเดียวได้**

*   เครื่องของคุณเริ่มเจอปัญหา "Generator Bottleneck" ตั้งแต่ตอนทดสอบ 2,000 VUs แล้ว
*   การพยายามรัน 5,000 VUs หรือมากกว่านั้นบนเครื่องเดียว จะทำให้ผลลัพธ์ที่ได้ **ไม่น่าเชื่อถืออย่างยิ่ง** เพราะคอขวดจะอยู่ที่โปรแกรม k6 ของคุณเอง ไม่ใช่ที่ NestJS Server

### วิธีทดสอบที่ถูกต้องสำหรับเป้าหมาย RPS สูงๆ

เราต้องเปลี่ยนวิธีการทดสอบจากการกำหนด VUs คงที่ ไปเป็นการกำหนด **อัตราการเกิด request (Arrival Rate)** ซึ่งจำลองสถานการณ์จริงได้ดีกว่า และใช้ Distributed Load Testing เพื่อสร้างโหลดให้ถึงเป้าหมาย

**1. แก้ไขสคริปต์ `tests/load-test/thundering-herd-problem.js`:**
เปลี่ยน `options` block เพื่อใช้ `constant-arrival-rate` executor นี่คือตัวอย่างสำหรับการทดสอบที่ 2,500 RPS

**ไฟล์: `tests/load-test/thundering-herd-problem.js` (ฉบับแก้ไข)**
```javascript
import http from 'k6/http';
// ... (imports อื่นๆ เหมือนเดิม) ...

export let options = {
  scenarios: {
    // กำหนดชื่อ scenario
    target_rps_test: {
      executor: 'constant-arrival-rate', // --- ใช้ executor ที่เน้น RPS ---

      // --- Configuration for 2,500 RPS ---
      rate: 2500,      // พยายามสร้าง request ใหม่ 2,500 ครั้ง
      timeUnit: '1s',    // ในทุกๆ 1 วินาที
      duration: '5m',    // ทดสอบเป็นเวลา 5 นาที
      preAllocatedVUs: 500, // เตรียม VUs ไว้ล่วงหน้า
      maxVUs: 4000,      // และให้ k6 สร้าง VUs เพิ่มได้สูงสุดตามต้องการเพื่อไปให้ถึงเป้าหมาย
    },
  },
  thresholds: {
    http_req_duration: ['p(95)<800'],
    errors: ['rate<0.05'],
  },
};

// --- ส่วนที่เหลือของไฟล์ (setup, default function, teardown) เหมือนเดิมทั้งหมด ---
// ...
```

**2. รันการทดสอบแบบกระจาย (Distributed Execution):**
คุณต้องใช้บริการอย่าง **Grafana Cloud k6** หรือตั้งค่า k6 cluster ของคุณเองบน Cloud (เช่น AWS/GCP) เพื่อรันสคริปต์นี้จากหลายๆ เครื่องพร้อมกัน เพื่อให้สามารถสร้างโหลดได้ถึง 2,500 RPS หรือสูงกว่านั้นได้จริง

**ขั้นตอนต่อไป:**
1.  อัปเดตสคริปต์ทดสอบของคุณตามตัวอย่างด้านบน (เริ่มที่ `rate: 2500`)
2.  ศึกษาและเตรียมพร้อมสำหรับการทำ Distributed Load Testing
3.  เมื่อพร้อมแล้ว ให้เริ่มทดสอบที่เป้าหมาย 2,500 RPS ก่อน แล้วค่อยๆ เพิ่ม `rate` ไปที่ 5000, 7500, และ 10000 ตามลำดับครับ

*   **เป้าหมายใหม่:** "จำลอง **ภาระงาน** ที่ X Requests Per Second (RPS) แล้วดูว่าระบบของเรารับไหวหรือไม่"

Executor ทั้งสองแบบถูกออกแบบมาเพื่อเป้าหมายที่แตกต่างกันโดยสิ้นเชิงครับ ผมจะอธิบายด้วยการเปรียบเทียบให้เห็นภาพชัดเจน

### เปรียบเทียบ Executor เหมือนแคชเชียร์ในซูเปอร์มาร์เก็ต

ลองจินตนาการว่า NestJS application ของเราคือ "ซูเปอร์มาร์เก็ต" ครับ

#### 1. `looping-vus` Executor (สิ่งที่เราทำมาตลอด)
*   **เปรียบเทียบ:** เหมือนการบอกว่า **"วันนี้ให้มีแคชเชียร์ทำงาน 1,000 คน (`vus: 1000`)"**
*   **การทำงาน:** แคชเชียร์แต่ละคนจะพยายามคิดเงินให้ลูกค้า (ประมวลผล 1 request) ให้เร็วที่สุดเท่าที่จะทำได้ พอเสร็จแล้วก็จะรับลูกค้ารายต่อไปทันที (เริ่ม iteration ใหม่)
*   **สิ่งที่เราวัดผล:** เราจะวัดว่า "ใน 1 วินาที แคชเชียร์ทั้ง 1,000 คน สามารถคิดเงินให้ลูกค้าได้ทั้งหมดกี่คน (RPS)"
*   **ข้อจำกัด:** **เราควบคุม RPS โดยตรงไม่ได้** ถ้าลูกค้าแต่ละคนมีของเยอะ (API response ช้า) จำนวนลูกค้าที่คิดเงินเสร็จใน 1 วินาที (RPS) ก็จะลดลงโดยอัตโนมัติ เราบอกได้แค่ว่า "ด้วยแคชเชียร์ 1,000 คน เราทำ RPS ได้สูงสุดเท่านี้"

#### 2. `constant-arrival-rate` Executor (สิ่งที่เรากำลังจะทำ)
*   **เปรียบเทียบ:** เหมือนการตั้งเป้าหมายว่า **"เราต้องรับลูกค้าใหม่ที่เดินมาถึงเคาน์เตอร์ให้ได้ 10,000 คนต่อวินาที (`rate: 10000`, `timeUnit: '1s'`)**"
*   **การทำงาน:** k6 จะพยายาม **เริ่มต้น** การคิดเงินให้ลูกค้ารายใหม่ตามอัตราที่เรากำหนด มันจะเริ่มจากมีแคชเชียร์จำนวนหนึ่ง (`preAllocatedVUs`) และถ้าเห็นว่าคิวยาวเกินไป มันจะ "จ้างแคชเชียร์เพิ่ม" ทันที (เพิ่มจำนวน VUs ขึ้นไปจนถึง `maxVUs`) เพื่อพยายามรักษาระดับ 10,000 RPS ให้ได้
*   **สิ่งที่เราวัดผล:** เราจะวัดว่า "ระบบของเราสามารถหาแคชเชียร์มาบริการลูกค้าใหม่ 10,000 คนต่อวินาทีได้ทันเวลาหรือไม่" ถ้าไม่ทัน k6 จะรายงานว่ามี `dropped_iterations` (ลูกค้าทนรอไม่ไหวและเดินออกจากร้านไป)
*   **ข้อดี:** **นี่คือการทดสอบ RPS ที่แท้จริง** มันจำลองสถานการณ์ที่ traffic เข้ามาอย่างต่อเนื่องตามที่เราต้องการ และทดสอบว่าระบบของเรา "เร็วพอ" ที่จะรองรับอัตรานั้นได้หรือไม่
